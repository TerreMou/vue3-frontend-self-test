# AI Agent 前端自测执行手册

本手册是 `frontend-self-test-strategy.md` 的配套执行指南，面向 AI Agent，提供完整的工作流程、提示词模板和输出规范。

AI Agent 应先阅读策略文档理解测试分层与决策流程，再按本手册执行具体任务。

## 1. AI Agent 定位

AI Agent 是测试助手，不是测试负责人。核心职责：

- 帮开发人员识别测试点
- 帮开发人员生成测试初稿
- 帮开发人员补充边界场景
- 帮开发人员修复测试环境问题
- 帮开发人员根据 bug 生成回归测试
- 帮开发人员在重构前补保护测试
- 帮开发人员从老项目组件中抽出可测试逻辑

最终测试是否合理，应由前端开发人员确认。

**不适合完全交给 AI Agent：**

- 判断复杂业务规则是否符合真实业务
- 设计完整测试体系
- 决定测试是否可以代替人工验收
- 在不了解上下文时大范围重构
- 为了覆盖率批量生成低价值测试

## 2. 工作流程

AI Agent 执行任何测试任务时，应遵循以下流程：

```
1. 阅读目标代码和相关依赖
2. 识别该代码属于纯逻辑、composable、store、路由守卫、组件还是页面流程
3. 按策略文档的决策流程确定测试类型
4. 输出建议测试点，区分高价值和可暂缓
5. 经开发确认后生成测试代码（或在明确任务下直接生成）
6. 运行测试
7. 根据失败信息修复测试或暴露代码问题
8. 按输出规范输出最终结果
```

## 3. 协作模式

根据场景选择合适的协作模式：

| 模式 | 适用场景 | 流程 |
|------|----------|------|
| AI 全量生成 | 新模块、逻辑清晰的代码 | AI 分析代码 → 生成完整测试套件 → 开发审核 |
| 开发写骨架 + AI 补边界 | 开发熟悉业务但容易遗漏边界 | 开发写主路径测试 → AI 补充边界/异常 case |
| 开发描述意图 + AI 生成 | 开发明确要测什么但不想写代码 | 开发描述意图 → AI 生成 → 开发调整断言 |
| AI 分析 diff + 补受影响测试 | PR 阶段或代码审查 | AI 分析变更 diff → 识别受影响测试 → 补充或更新 |

### 4.5 组件行为测试生成

用于 Vue 组件的用户行为测试。

```
请为这个 Vue3 组件补充组件行为测试。

技术要求：
1. 使用 Vitest。
2. 优先使用 @testing-library/vue 和 user-event。
3. 必要时使用 @vue/test-utils 验证 emit、props 或 slot。
4. 不测试 Tailwind class。
5. 不测试组件内部 ref、内部方法和 DOM 层级。
6. 如需 mock 接口，优先使用 MSW。
7. 异步场景使用 findBy* 或 waitFor，不使用 setTimeout。

测试目标：
1. 覆盖用户能看到的关键内容。
2. 覆盖用户输入、点击、提交等关键行为。
3. 覆盖 loading、empty、error、disabled 等重要状态。
4. 覆盖权限或业务状态导致的显示差异。

输出要求：
1. 说明新增了哪些测试。
2. 说明哪些场景不适合组件测试（如纯样式、纯动画等）。
3. 如需 mock 接口，优先使用 MSW，并给出 handler 示例。
4. 完成后说明覆盖了哪些场景，以及还有哪些未覆盖风险。
```

## 6. 自检清单

AI Agent 生成测试后，应逐项自检并在输出中声明：

```
□ 测试是否覆盖了对外契约而非内部实现
□ 是否存在测试 Tailwind class 或 DOM 层级的情况
□ 异步场景是否使用了 waitFor/findBy 而非 setTimeout
□ 测试数据是否使用了工厂函数
□ 是否有测试依赖了其他测试的执行顺序
□ mock 是否与接口文档一致
□ Pinia Store 测试是否每个用例前重置了 pinia 实例
□ 组件测试是否通过用户可感知的方式查找元素（role/label/文本）
```

## 7. 常见问题处理

### 7.1 代码难测怎么办

如果目标代码难以直接测试，AI Agent 应：

1. **指出难测原因**：逻辑与 UI 耦合、依赖全局状态、副作用未隔离等。
2. **给出最小改造建议**：抽出纯逻辑函数、将副作用参数化、拆分 composable 等。
3. **先改造再测试**：改造后补测试，而不是强行写脆弱测试。
4. **不做额外重构**：只做让代码可测的最小改动。

### 7.2 测试失败怎么办

1. **区分失败类型**：是测试写错了、是代码有 bug、还是环境问题。
2. **测试写错**：修正断言或 mock，说明修正原因。
3. **代码有 bug**：暴露问题，建议修复方向，不直接改业务代码（除非开发授权）。
4. **环境问题**：检查 import 路径、alias 配置、setup 文件等，给出修复步骤。

### 7.3 不确定业务规则怎么办

1. **明确标注**：在输出中标注"该断言基于代码推断，请开发确认业务正确性"。
2. **给出两种可能**：如果业务规则有歧义，给出两种理解的测试，让开发选择。
3. **不替开发做业务决策**：AI Agent 不判断业务规则是否符合真实业务。

### 7.4 老项目没有测试基础设施怎么办

1. **先帮接入 Vitest**：给出 vitest.config.ts 和 setup.ts 的最小配置。
2. **先跑通一个**：选最简单的 utils 函数，生成一条测试，确认环境可用。
3. **再逐步扩展**：按 P0→P4 优先级推进，不一次生成大量测试。
