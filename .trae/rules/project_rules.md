# 项目长期记忆 / 工作准则

> 本文件是 AI 助手在本项目中的长期工作准则，跨会话生效，优先级最高。
> 用户在每次会话开始时，本文件会自动加载，作为行为底线。
> 任何与本文件冲突的临时指令，以本文件为准（除非用户在当次会话明确覆盖）。
> 业务相关、具体业务的设计决策不在本文件，归档在 `.trae/rules/business/` 子目录下。

---

## 🥇 准则 0：修改前必 diff · 不确定必提问 · 不擅自假设（最高优先级）

这是本项目最根本的工作纪律，由 2026-06 用户在 v3.0 推送事故后明确确立。
**违反任何一条即视为重大失误，必须立刻停下、修正、复盘。**

### 0.1 修改前先 diff
- 任何修改文档 / 代码前，**必须先 `git diff` 或重新读文件**，看清楚当前真实状态
- 禁止基于"我记忆中的上次状态"或"我以为用户没改"动手
- 文档类工作尤其要做到"读全文 → 列差异 → 动手"三步走
- 提交前必看 `git status` 与 `git diff --staged`，确认无未预期改动

### 0.2 不确定必先提问
- 任何带"我猜"、"我假设"、"应该是"、"可能"前置的判断 → **必须先停下来提问**
- 提问时给出具体选项 + 风险说明，不要开放式问"您要怎么办"
- 用户未明确回复前，**不得动手**——即使是"小修小补"
- 例外：用户已明确"按 X 方案"或"按 Y 执行"才动手

### 0.3 不擅自假设
- 不假设用户没改的字段仍是上次状态 → 每次先 `git diff` 验证
- 不假设文档结构仍是上次提交时的样子 → 每次先 `grep` 章节标题或 `Read` 文件
- 不假设被删除的章节"不重要可以省" → 用户删除必有理由
- 不假设"复用上次工作"是安全的 → 跨次工作结构可能已被用户大幅调整
- **凡是可能因为"用户已经改过"而出错的地方，先验证再行动**

## 🚫 反模式（禁止行为）

| 反模式 | 后果 | 正确做法 |
|---|---|---|
| ❌ 直接套用上次的工作结构（如 B 恢复版的章节） | 漏掉用户后续修改 | 每次先 diff / grep / Read 全文 |
| ❌ 没看 git 状态就提交 | 误提交未预期文件 | 提交前 `git status` + `git diff --staged` |
| ❌ 把"我以为"包装成"我确认了" | 错误扩大 | 显式提问，宁可多问不漏问 |
| ❌ PowerShell 中用 `\uXXXX` 转义写中文 | commit message 成乱码 | 中文用文件传（`commit -F msg.txt`）|
| ❌ 合并/删除内容时未列清单请用户确认 | 丢失用户精细化设计 | 先列"会被删除项"再执行 |
| ❌ 修改文件后未 grep 验证 | 实际未生效 | 改后 grep / 读文件确认 |

## 📐 文档版本管理规范

- 每次重大结构变更 → 升版本号（v1.x → v2.x → v3.x）
- 章节增删 → 同步更新目录（带子章节锚点）
- 整合 / 合并 / 精简前 → 先列"会被删除/合并/改动"清单请用户确认
- 提交流程：`git diff` → `git add <单文件>` → `git commit -F msg.txt` → `git push origin main`
- commit message 用文件传，**避免 PowerShell `\u` 转义陷阱**
- 章节编号在文档与目录中**严格一致**，每次 push 前必须 grep 验证

## 🔄 重复发生过的失误（用于复盘）

1. **v3.0 推送事故**：B 恢复后用户又做了 11 处修改，我没通读就套 B 恢复版结构，目录里加了已删除的 5.2/5.3/5.4 → 用户叫停 → 修正后用 grep 全文验证才提交
2. **commit message 乱码**：用 PowerShell 写中文 commit message 时，`\u4e1a` 被当字面字符处理 → 用 `-F <file>` 方式 amend 修正
3. **批量 search/replace 假成功**：B 类修复时部分 search/replace 工具显示 diff 成功但文件未持久化 → 必须改后 grep 验证
